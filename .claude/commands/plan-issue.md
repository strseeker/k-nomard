# GitHub 이슈 플래닝 커맨드

## 사용방법

```
/plan-issue <이슈 번호>
```

**예시:**
```
/plan-issue 12
/plan-issue 34
```

---

## 프롬프트

당신은 K·NOMADS 프로젝트의 시니어 엔지니어입니다.
GitHub 이슈를 분석하고 구현 계획을 수립한 뒤, **반드시 사용자의 승인을 받고** 작업을 시작합니다.

**이슈 번호:** $ARGUMENTS

---

### 1단계: 이슈 가져오기

아래 명령어로 이슈 전체 내용을 가져오세요:

```bash
gh issue view $ARGUMENTS --json number,title,body,labels,assignees,milestone,comments
```

이슈 내용을 읽고 다음 정보를 파악하세요:
- 이슈 제목 및 유형 (버그 / 기능 / 개선)
- 요구 사항 및 완료 조건
- 관련 라벨 및 컨텍스트
- 댓글에서 추가 논의된 내용

---

### 2단계: 코드베이스 분석

이슈와 관련된 코드를 파악하세요:
- 영향받는 파일 및 컴포넌트 탐색
- 기존 패턴 및 아키텍처 확인
- 의존성 및 연관 모듈 파악

---

### 3단계: 구현 계획 작성

아래 형식으로 상세한 구현 계획을 작성하세요:

---

> (이슈를 한 문장으로 요약)

### 원인 분석 (버그인 경우)

### 구현 접근 방식
> (어떤 방식으로 해결할 것인지 기술적 방향 설명)

### 완료 조건 확인

- [ ] 이슈에서 요구한 기능/수정이 동작함
- [ ] 기존 기능이 깨지지 않음
- [ ] TypeScript 타입 오류 없음 (`npx tsc --noEmit`)
- [ ] 모바일/데스크탑 반응형 확인 (UI 변경 시)

### 4단계: 사용자 검토 요청

계획 작성이 완료되면 **반드시** 아래 메시지로 사용자에게 검토를 요청하고 대기하세요:

---

> **위 계획을 검토해 주세요.**
>
> - 계획대로 진행할까요?
> - 수정이 필요한 부분이 있으면 알려주세요.
> - 승인해 주시면 바로 구현을 시작하겠습니다.

**사용자가 승인하기 전까지는 절대로 코드를 수정하거나 파일을 변경하지 마세요.**
승인 후 작업 1번부터 순서대로 구현을 시작하세요.
